# pi0.5 模型与代码入门：从一条 RoboTwin 样本到 50 步动作

本文面向第一次接触 pi0.5、希望通过代码理解模型的读者。内容以本仓库
`policy/pi05` 下的 JAX/openpi 实现和当前任务 `adjust_bottle_singlearm` 为准。

建议不要一开始逐行阅读整个仓库。先记住一条主线：

> pi0.5 把相机图像、任务文字和离散化的机器人状态作为条件，把一整段带噪动作交给
> action expert，并学习一个将噪声“运回”真实动作的速度场。

## 1. 论文中的 pi0.5 与本仓库实现的边界

[pi0.5 论文](https://arxiv.org/abs/2504.16054)介绍的是一个面向开放世界泛化的完整系统。
完整系统可以先生成高层语言子任务，再生成低层连续动作，并通过多来源数据共同训练。

不过，[openpi 官方仓库说明](https://github.com/Physical-Intelligence/openpi)明确指出，当前开源的
pi0.5 训练与推理只支持 **flow matching 动作头**。因此，在这份 RoboTwin 代码中：

- 任务语言和离散机器人状态是模型输入；
- 模型输出连续动作块；
- 不会生成高层子任务文本；
- `compute_loss()` 只有连续动作的 flow-matching loss，没有高层文本生成 loss。

理解论文时可以关注“开放世界、多源数据、语义泛化”；理解本地训练代码时，应把注意力放在
“多模态条件 + action expert + flow matching”上。

## 2. 先建立整体结构

一条训练样本的路径如下：

```text
LeRobot 当前帧
  ├─ 3 路 RGB 图像 ── resize/增强 ── SigLIP ───────────────┐
  ├─ task prompt ─────────────────── SentencePiece ────────┤ prefix
  ├─ 14 维 state ── 归一化 ── 256 桶离散化 ── 写入 prompt ─┘
  │
  └─ 未来 50 帧 action
       └─ 关节改成相对动作 ── 归一化 ── 14→32 维补零
            └─ 与高斯噪声插值得到 x_t ── action projection ─┐ suffix
                                                           │
 prefix ── PaliGemma expert ───────┐                       │
                                   ├─ 联合 Transformer attention
 suffix ── action expert + adaRMS ─┘
                    │
                    └─ 预测速度 v_t，与 noise - action 做 MSE
```

推理时没有真实 action。模型从随机高斯噪声开始，默认用 10 次 Euler 更新把它变成完整的
50 步动作块。

## 3. 当前配置中最重要的尺寸

训练配置位于：

```text
policy/pi05/src/openpi/training/config.py
└─ pi05_base_adjust_bottle_singlearm_lora
```

当前主要尺寸如下：

| 项目 | 形状或配置 | 含义 |
| --- | --- | --- |
| 图像 | 3 × `[B, 224, 224, 3]` | 头部、左腕、右腕相机 |
| 原始 state | `[B, 14]` | 两臂关节与夹爪状态 |
| 模型 state | `[B, 32]` | 统一接口补零后的形状 |
| prompt token | `[B, 200]` | 包含任务文字和离散 state |
| 原始 action chunk | `[B, 50, 14]` | 当前帧开始的未来 50 帧动作 |
| 模型 action chunk | `[B, 50, 32]` | 补零后送入 action expert |
| PaliGemma expert | Gemma 2B | 理解图像和文字条件 |
| action expert | Gemma 300M | 处理带噪动作并预测速度 |
| 数值类型 | bfloat16 为主 | 冻结参数会转为 bfloat16 |

SigLIP 的 patch 大小为 14。每张 224×224 图像产生 16×16=256 个图像 token，三路相机
共 768 个图像 token；再加最多 200 个 prompt token 和 50 个 action token，一次前向的
最大序列规模约为 1018 个 token。padding token 会被 attention mask 屏蔽。

## 4. 一条样本如何进入模型

### 4.1 LeRobot 读取未来动作块

从这里开始读：

```text
policy/pi05/src/openpi/training/data_loader.py
└─ create_torch_dataset()
```

核心代码根据数据集 FPS 构造未来时间偏移：

```python
delta_timestamps={
    key: [t / dataset_meta.fps for t in range(action_horizon)]
    for key in data_config.action_sequence_keys
}
```

`action_horizon=50`。当前数据集为 50 FPS 时，一个 action chunk 覆盖约 1 秒。每个训练样本
不是只监督当前动作，而是监督从当前帧开始的 50 个动作。

数据变换严格按以下顺序执行：

```text
repack_transforms
→ data_transforms
→ Normalize
→ model_transforms
```

顺序很重要。例如 pi0.5 是先归一化 state，之后才把它离散化为 prompt 中的桶编号。

### 4.2 Repack：把数据集字段改成策略认识的字段

当前配置中的映射为：

```python
{
    "images": {
        "cam_high": "observation.images.cam_high",
        "cam_left_wrist": "observation.images.cam_left_wrist",
        "cam_right_wrist": "observation.images.cam_right_wrist",
    },
    "state": "observation.state",
    "actions": "action",
    "prompt": "prompt",
}
```

这一步只改 key，不改变数据含义。

### 4.3 ALOHA 变换与相对关节动作

`LeRobotAlohaDataConfig` 默认启用 `use_delta_joint_actions=True`。对应 mask 是：

```python
make_bool_mask(6, -1, 6, -1)
```

14 个动作维度的语义可以理解为：

```text
[左臂 6 个关节, 左夹爪, 右臂 6 个关节, 右夹爪]
```

12 个关节维度会转换成相对当前 state 的 delta，两个夹爪维度仍保持绝对值。推理输出时，
`AbsoluteActions` 再执行逆变换。

当前配置使用 `adapt_to_pi=False`，因此不会额外应用 Physical Intelligence 内部 ALOHA
坐标约定的关节翻转或夹爪转换。这与当前 RoboTwin 数据格式相匹配。

### 4.4 归一化

pi0.5 配置会使用 quantile normalization。统计量来自：

```text
policy/pi05/assets/pi05_base_adjust_bottle_singlearm_lora/
└─ adjust_bottle_singlearm/norm_stats.json
```

训练和推理必须使用同一份统计量。checkpoint 保存时会把 assets 一起保存，加载策略时优先
从 checkpoint 读取它们，防止训练、推理的数值空间不一致。

### 4.5 pi0.5 如何把 state 写进 prompt

阅读：

```text
policy/pi05/src/openpi/models/tokenizer.py
└─ PaligemmaTokenizer.tokenize()
```

归一化后的每一维 state 会在 `[-1, 1]` 上划分到 256 个桶，再构造成类似文本：

```text
Task: adjust bottle, State: 121 133 98 ...;
Action:
```

之后整个字符串再由 PaliGemma 的 SentencePiece tokenizer 编码，并 padding/truncate 到 200。

需要注意两个细节：

1. 写进 prompt 的是补零前的 14 维归一化 state，因为 `TokenizePrompt` 在
   `PadStatesAndActions` 之前执行。
2. `Observation.state` 仍然存在并被补到 32 维，但 pi0.5 的 `embed_suffix()` 不会再把它
   投影成连续 state token。它仍用于接口、batch size 和动作逆变换等流程。

quantile normalization 会把训练数据的 1% 和 99% 分位点映射到约 `-1` 和 `1`，但代码没有
执行 clip。因此少量离群值仍可能越界：低端可能得到桶编号 `-1`，高端会进入 `255`。调试数据
异常时，应同时检查归一化值和离散桶编号。

### 4.6 14 维为什么变成 32 维

`PadStatesAndActions(32)` 把 state 和 action 的最后一维从 14 补零到 32。这样不同机器人
可以复用固定宽度的动作接口。模型输出后，`AlohaOutputs` 只取前 14 维。

这 18 个 padding 维度同样参与训练 loss，但监督目标始终是零。它们不是额外的机器人自由度。

## 5. 模型主体：两个 expert 如何合作

主要文件：

```text
policy/pi05/src/openpi/models/pi0.py
policy/pi05/src/openpi/models/gemma.py
policy/pi05/src/openpi/models/siglip.py
```

### 5.1 PaliGemma expert

PaliGemma 侧包括：

- SigLIP 视觉编码器：把三路图像转换成 patch token；
- Gemma 2B：处理图像 token、任务文字 token 和离散 state token；
- 宽度 2048、18 层、8 个 attention head。

这部分形成条件信息，也就是 `prefix`。

### 5.2 action expert

action expert 使用较小的 Gemma 300M 配置：

- 宽度 1024；
- 18 层；
- 8 个 attention head；
- 输入是 `[B, 50, 32]` 的带噪动作块经过 `action_in_proj` 后的 token；
- 输出经过 `action_out_proj` 恢复成 `[B, 50, 32]` 的速度预测。

它不是一个简单 MLP，而是与 PaliGemma 同层运行的 Transformer expert。两个 expert 有各自的
FFN、RMSNorm 等参数，但 attention 会组合它们产生的 query/key/value，从而让动作 token 读取
图像和语言条件。

### 5.3 prefix 与 suffix 的注意力规则

`embed_prefix()` 构造图像和语言 token，`embed_suffix()` 构造动作 token。

当前 pi0.5 的 mask 可简化成：

```text
                 被读取的 token
查询 token       prefix       action chunk
------------------------------------------------
prefix             可以            不可以
action chunk       可以            可以（块内双向）
```

因此动作 expert 能根据图像、指令和 state 预测动作；条件 token 不会被当前带噪动作污染。50 个
动作 token 在同一个 attention block 中，能互相双向注意，所以这是整块并行去噪，不是按时间
自回归地生成 50 次。

实现 mask 的关键函数是 `make_attn_mask()`：它对 `mask_ar` 做累计和，把 token 划分为若干
block，再允许每个 block 读取自己和之前的 block。

## 6. pi0.5 相比 pi0 改了什么

本仓库让 pi0 和 pi0.5 共用 `Pi0` 类，通过 `Pi0Config(pi05=True)` 切换分支。

| 位置 | pi0 | pi0.5 |
| --- | --- | --- |
| state 输入 | 连续 state 投影成一个 suffix token | state 离散成 256 桶并写进语言 prompt |
| prompt 最大长度 | 48 | 200 |
| flow timestep | 与每个 action token 拼接后过 MLP | 经 MLP 后作为 adaptive RMSNorm 条件 |
| action 输出 | flow matching | flow matching |

决定这些差异的入口是：

```text
pi0_config.py
├─ pi05
├─ max_token_len
└─ discrete_state_input

pi0.py
├─ __init__()       # 是否创建 adaptive RMSNorm 的时间 MLP
├─ embed_suffix()   # state 与 timestep 的不同注入方式
├─ compute_loss()   # 两者共用 flow-matching 训练目标
└─ sample_actions() # 两者共用数值积分采样
```

## 7. adaptive RMSNorm 在做什么

阅读：

```text
policy/pi05/src/openpi/models/gemma.py
├─ RMSNorm
└─ _gated_residual
```

普通 RMSNorm 只根据当前 token 激活做归一化。adaptive RMSNorm 还接收 flow timestep 的条件
向量，并生成 scale、shift 和 gate，用它调节 action expert 每层的归一化和残差强度。

直观理解：同一个带噪动作在“几乎全是噪声”和“已经接近真实动作”两个阶段，需要不同的处理
策略。时间条件告诉每一层当前处于去噪过程的哪个位置。

pi0.5 中的路径是：

```text
t
→ sin/cos embedding
→ time_mlp_in
→ SiLU
→ time_mlp_out
→ SiLU
→ action expert 每层的 adaptive RMSNorm
```

## 8. Flow matching：训练究竟在拟合什么

核心代码在 `Pi0.compute_loss()`。

给定数据中的真实动作块 `a` 和同形状高斯噪声 `ε`，随机采样 `t∈(0,1]`：

```text
x_t = t ε + (1 - t) a
u_t = ε - a
```

这里采用的约定是：

- `t=0`：真实动作；
- `t=1`：纯噪声；
- `x_t`：二者之间的直线插值；
- `u_t`：这条直线路径的目标速度。

模型接收条件信息、`x_t` 和 `t`，输出预测速度 `v_t`。损失是：

```text
loss = mean((v_t - u_t)²)
```

代码先在 32 个动作维度上取平均，得到 `[B, 50]`，训练循环再取总平均得到标量。

这里的“去噪”不是让模型直接猜出 `a`，而是学习每个位置应该往哪个方向移动。只要速度场学得
足够准确，推理时就可以从随机噪声沿速度场积分到动作分布。

## 9. 推理为什么要迭代 10 次

核心代码在 `Pi0.sample_actions()`。

算法可以简化成：

```python
x_t = gaussian_noise
t = 1.0
dt = -1.0 / num_steps

while t > 0:
    v_t = model(prefix, x_t, t)
    x_t = x_t + dt * v_t
    t = t + dt

return x_t
```

默认 `num_steps=10`，即使用显式 Euler 法从 `t=1` 积分到 `t=0`。每次迭代都处理完整的
50×32 动作张量。

图像和 prompt 构成的 prefix 在 10 次迭代中不变，所以代码先计算一次 prefix 并保存 KV
cache，后续只重算动作 suffix。这是推理速度的重要优化。

最终输出还要依次执行：

```text
反归一化前 14 个真实动作维度（padding 尾部保持不变）
→ 相对关节动作恢复为绝对关节动作
→ AlohaOutputs 取前 14 维
→ 返回 [50, 14] action chunk
```

RoboTwin 的 `policy/pi05/pi_model.py` 再负责把仿真器的三路图像、state 和 instruction 包装成
策略输入。

## 10. LoRA 微调时到底训练了哪些参数

当前配置使用：

```python
paligemma_variant="gemma_2b_lora"
action_expert_variant="gemma_300m_lora"
```

对应 LoRA 设置为：

- Gemma 2B attention/FFN：rank 16，alpha 16；
- action expert attention/FFN：rank 32，alpha 32。

LoRA 在线性映射旁增加低秩更新：

```text
y = W x + scale · B(Ax)
scale = alpha / rank
```

实现位于 `policy/pi05/src/openpi/models/lora.py`。

一个非常重要的代码事实是：本配置的 freeze filter 只冻结 Gemma 主干中非 LoRA 参数。视觉
编码器、动作输入/输出投影、time MLP 等仍然可训练。因此这里的“LoRA 微调”不等同于“全模型
只有 LoRA 矩阵在更新”。

按当前配置创建参数树得到：

| 参数 | 数量 |
| --- | ---: |
| 总参数 | 3,403,421,456 |
| 可训练参数 | 466,957,072 |
| 冻结参数 | 2,936,464,384 |
| 可训练占比 | 约 13.72% |

如果之后需要进一步降低显存，不能只继续降低 LoRA rank；还需要考虑是否冻结 SigLIP 或其他
投影层。但这会改变微调策略，应通过验证集或仿真成功率判断收益与损失。

## 11. 训练循环如何工作

从 `policy/pi05/scripts/train.py` 阅读。

### 11.1 初始化

`init_train_state()` 完成：

1. 创建模型结构与随机初始化参数；
2. 从 `pi05_base` checkpoint 加载匹配的预训练参数；
3. 按 freeze filter 把冻结参数转成 bfloat16；
4. 只为可训练参数创建 AdamW optimizer state；
5. 创建 EMA 参数副本；
6. 根据双卡 FSDP mesh 对参数和 optimizer state 做 sharding。

### 11.2 单个训练 step

`train_step()` 的逻辑是：

```text
model.compute_loss()
→ 只对 trainable_filter 求梯度
→ AdamW 更新可训练参数
→ EMA = 0.99 × 旧 EMA + 0.01 × 新参数
→ 记录 loss / grad_norm / param_norm
```

推理加载 checkpoint 时默认使用 EMA 参数。EMA 往往比某一步瞬时参数更稳定。

### 11.3 JIT 与双卡 FSDP

`fsdp_devices=2` 会把两张 3090 组成一个 FSDP mesh。全局 batch 会按 data axis 分发，较大的
参数和 optimizer state 也会按规则分片。

第一次执行训练 step 时 JAX 需要编译完整计算图，等待较久属于正常现象；之后 step 才进入
稳定速度。全局 batch 必须能被设备数整除。

当前优化器默认值可在 `training/optimizer.py` 查看：

- cosine learning-rate schedule；
- warmup 1000 steps；
- peak learning rate `2.5e-5`；
- AdamW，`β1=0.9`，`β2=0.95`；
- global gradient clipping 为 1.0。

## 12. 当前任务的训练命令

在 `policy/pi05` 目录中：

```bash
CUDA_VISIBLE_DEVICES=0,1 \
XLA_PYTHON_CLIENT_PREALLOCATE=false \
uv run scripts/train.py \
  pi05_base_adjust_bottle_singlearm_lora \
  --exp-name=adjust_bottle_singlearm_lora \
  --overwrite \
  --batch-size=16
```

配置文件默认全局 batch 为 32，上面的命令显式改为 16，适合先在两张 24 GB 3090 上验证稳定
运行。如果显存和吞吐允许，可以去掉 `--batch-size=16` 使用配置值 32。

`--overwrite` 会覆盖同名实验目录，只适合重新开始训练。需要从已有 checkpoint 接着训练时，
应改用对应的 resume 选项，不要同时覆盖目录。

## 14. 适合入门的代码实验

不必马上改模型，可以先做以下只读实验：

### 实验一：打印一批数据的 shape

在 data loader 初始化后的日志中确认：

```text
images/*          [B, 224, 224, 3]
state             [B, 32]
tokenized_prompt  [B, 200]
actions           [B, 50, 32]
```

若这里不正确，问题仍在数据层，不要先调模型。

### 实验二：观察 state 离散化

选一条样本，分别打印归一化 state、256 桶编号和 `full_prompt`。这能直接建立“连续机器人状态
如何成为语言输入”的直觉。

### 实验三：固定推理噪声

`Policy.infer(obs, noise=...)` 支持传入固定噪声。对相同 observation 使用同一份 noise，便于
比较不同 checkpoint 或不同 `num_steps`，排除随机起点的干扰。

### 实验四：观察 flow 插值

对同一个 action 和 noise，取 `t=0、0.25、0.5、0.75、1`，画出某个关节的 `x_t`。这样可以
直观看到模型训练时面对的输入从真实动作逐渐过渡到噪声。

## 15. 常见误解和排查顺序

### “pi0.5 是语言模型，所以动作也是文字 token”

不是当前这条实现。state 被离散成文字输入，但 action 仍是连续值，由 flow-matching head 输出。

### “50 个动作是一个接一个自回归生成的”

不是。50 个 action token 在同一个 attention block 内双向可见，并行经历每次去噪更新。

### “启用 LoRA 后只有很少参数训练”

不完全是。Gemma 主干被冻结，但视觉编码器和动作相关层仍可训练；当前可训练参数约 4.67 亿。

### “训练 loss 下降就说明仿真任务一定成功”

不成立。loss 衡量的是归一化动作空间里的速度拟合。最终还要检查动作反归一化、delta/absolute
转换、左右相机映射、控制频率，并通过 RoboTwin rollout 统计任务成功率。

### 出现异常时按什么顺序查

```text
数据 key 与 shape
→ 相机顺序和颜色格式
→ prompt 是否正确
→ norm_stats 是否对应当前数据集
→ delta action / absolute action 是否成对
→ checkpoint 是否加载了同一训练配置
→ loss、grad_norm 是否有限
→ 仿真控制频率与 action chunk 执行方式
```

## 16. 术语速查

| 术语 | 本文中的含义 |
| --- | --- |
| VLA | 输入视觉与语言、输出机器人动作的模型 |
| prefix | 图像、任务文字和离散 state 等条件 token |
| suffix | 当前带噪的整段 action token |
| action chunk | 一次预测的连续多步动作，当前为 50 步 |
| flow matching | 学习从数据分布到噪声分布路径上的速度场 |
| LoRA | 在线性层旁增加可训练低秩更新 |
| adaptive RMSNorm | 使用 timestep 条件调制 action expert 的归一化与残差 |
| KV cache | 缓存不变 prefix 的 attention key/value，避免每次去噪重复计算 |
| EMA | 对训练参数做指数滑动平均，用作更稳定的推理参数 |
| FSDP | 将参数、状态和计算按设备 mesh 分片 |
